# go-events

A multi-tenant event management API built with **Go + chi + pgx**, whose schema is fully managed by pg-flux.

This example exists to show what a Go service looks like when pg-flux owns the schema and the app uses `pg-flux gen --lang go` for its database types. The schema grew across 15 real iterations — covering `GENERATED ALWAYS AS IDENTITY` PKs, `DEFERRABLE INITIALLY DEFERRED` FKs, a materialized view, GIN indexes, RLS, and a soft-delete pattern. See `JOURNEY.md` for the unfiltered account of what happened along the way.

## Layout

```
go-events/
├── main.go                   # chi router, listen on :8013
├── go.mod
├── gen/                      # pg-flux gen --lang go output — DO NOT EDIT
│   ├── tables.go             # Workspace, Event, User, Attendee structs
│   ├── enums.go              # UserRole, EventStatus, AttendeeStatus constants
│   └── views.go              # EventStat (materialized view row)
├── internal/
│   ├── db/
│   │   └── conn.go           # pgx pool from DATABASE_URL
│   └── handler/
│       ├── workspaces.go     # POST/GET /workspaces
│       ├── events.go         # CRUD + stats under /workspaces/:slug/events
│       └── attendees.go      # POST/GET /events/:id/attendees
├── schema/                   # pg-flux source of truth — edit these
│   ├── types.sql             # user_role, event_status, attendee_status enums
│   ├── workspaces.sql        # GENERATED ALWAYS AS IDENTITY PK
│   ├── users.sql             # GENERATED ALWAYS AS IDENTITY, @renamed hint
│   ├── events.sql            # soft delete, GIN indexes, RLS, updated_at trigger
│   ├── attendees.sql         # DEFERRABLE INITIALLY DEFERRED FK
│   ├── views.sql             # CREATE MATERIALIZED VIEW event_stats
│   ├── functions.sql         # count_confirmed_attendees(event_id)
│   ├── rls.sql               # events workspace isolation policy
│   └── triggers.sql          # set_updated_at() + events_set_updated_at trigger
├── migrations/               # generated by pg-flux — review, don't hand-edit
├── .env.example
└── .pg-flux.yml
```

## Run it

```bash
# 1. Install pg-flux (no Go required)
curl -sSfL https://raw.githubusercontent.com/nex-gen-tech/pg-flux/main/install.sh | sh

# 2. Set up the database
cp .env.example .env       # edit if your DSN differs
PGPASSWORD=pgflux createdb -h localhost -p 5440 -U pgflux pgflux_go_events

# 3. Apply the schema
set -a && source .env && set +a
pg-flux migrate apply

# 4. Run the API
go run .
```

The server starts on port `8013`. Then in another terminal:

```bash
# Create a workspace
curl -sX POST http://localhost:8013/workspaces \
  -H 'Content-Type: application/json' \
  -d '{"slug":"acme","name":"Acme Corp","plan":"pro"}' | jq

# Get workspace by slug
curl -s http://localhost:8013/workspaces/acme | jq

# Create an event
curl -sX POST http://localhost:8013/workspaces/acme/events \
  -H 'Content-Type: application/json' \
  -d '{
    "title": "Q3 Kickoff",
    "description": "All-hands event",
    "starts_at": "2026-07-01T09:00:00Z",
    "ends_at": "2026-07-01T17:00:00Z",
    "location": "Main Hall",
    "capacity": 200
  }' | jq

# List events (filter by status)
curl -s "http://localhost:8013/workspaces/acme/events?status=draft" | jq

# Get event with attendee stats from materialized view
curl -s http://localhost:8013/workspaces/acme/events/1 | jq

# Publish the event
curl -sX PATCH http://localhost:8013/events/1 \
  -H 'Content-Type: application/json' \
  -d '{"status":"published"}' | jq

# Register an attendee (uses DEFERRABLE FK transaction)
curl -sX POST http://localhost:8013/events/1/attendees \
  -H 'Content-Type: application/json' \
  -d '{"user_id":1,"status":"confirmed"}' | jq

# Confirmed count via SQL function (not materialized view)
curl -s http://localhost:8013/_stats/events/1 | jq

# Soft-delete the event
curl -sX DELETE http://localhost:8013/events/1
```

## What the schema demonstrates

The schema was built up across 15 iterations — intentionally different from the previous two examples to exercise capabilities not yet covered.

| Schema feature | File | Why it's here |
|---|---|---|
| `GENERATED ALWAYS AS IDENTITY` PKs | all tables | PG15+ identity columns; different from bigserial and uuid |
| `DEFERRABLE INITIALLY DEFERRED` FK | `attendees.sql` | FK checked at transaction commit, not statement end |
| `UNIQUE NULLS NOT DISTINCT` approximation | `events.sql` (partial unique index) | One active event per title per workspace |
| `CREATE MATERIALIZED VIEW` | `views.sql` | Confirmed attendee counts; refreshed concurrently |
| `REFRESH MATERIALIZED VIEW CONCURRENTLY` | `attendees.go` | Non-blocking refresh after each registration |
| `pg-flux gen --lang go` | `gen/` | Generated structs used directly as pgx scan targets |
| Updated_at trigger | `triggers.sql` | DB-side `updated_at` via PL/pgSQL trigger |
| RLS with bigint setting | `rls.sql` | Workspace isolation via `current_setting('app.workspace_id')::bigint` |
| Soft-delete pattern | `events.deleted_at` | Partial unique index keeps active queries fast |
| `GENERATED ALWAYS AS ... STORED` | `events.sql` (`title_lower`) | Case-insensitive search without lower() in query |
| GIN index on jsonb | `events.sql` (`metadata`) | `@>` containment queries on event metadata |
| GIN index on text[] | `events.sql` (`tags`) | `@>` containment queries on tag arrays |
| `@renamed from=` hint | `users.sql` (username→display_name) | Data-safe column rename |
| Multiple enums | `types.sql` | `user_role`, `event_status`, `attendee_status` |

## The pg-flux workflow

```bash
# 1. Edit schema/*.sql
$EDITOR schema/events.sql

# 2. Generate the migration
pg-flux migrate generate --label my_change

# 3. Review the file in migrations/ — it's plain SQL
$EDITOR migrations/20260521_*.sql

# 4. Apply
pg-flux migrate apply
```

For CI:

```bash
pg-flux drift    # exit 1 if live DB != source (materialized views handled correctly — no false positives)
pg-flux verify   # exit 1 if live has undeclared objects (enums and materialized view handled correctly)
```

## Notes

- The structs in `gen/` are generated by `pg-flux gen --lang go`. Do not edit them by hand. To regenerate: `pg-flux gen --lang go --out gen/`.
- The `gen` package name is `dbgen`. Import it as `dbgen "github.com/nex-gen-tech/pg-flux/examples/go-events/gen"`.
- The enum types have Go names that differ from the SQL names: `event_status` → `EventStatu` (the codegen singularizes the last word; `status` → `statu`). See JOURNEY.md B1 for details.
- The app uses pgx directly — no ORM, no query builder. All SQL is in handler files.
- The `updated_at` column is maintained by the DB-side trigger `events_set_updated_at`, not by the app.
- After each attendee registration the app calls `REFRESH MATERIALIZED VIEW CONCURRENTLY public.event_stats`. In production, do this in a background job.
- `DATABASE_URL` is used by both pg-flux and the app.

## Reset to a clean DB

```bash
PGPASSWORD=pgflux dropdb   -h localhost -p 5440 -U pgflux pgflux_go_events
PGPASSWORD=pgflux createdb -h localhost -p 5440 -U pgflux pgflux_go_events
pg-flux migrate apply
```
