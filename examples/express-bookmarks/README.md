# express-bookmarks

A bookmarks / reading-list API built with **TypeScript + Express 4 + node-postgres**, whose schema is fully managed by pg-flux.

This example exists to show what an end-to-end stack looks like when pg-flux owns the schema and a Node.js app reads from it. The schema grew across ~18 real iterations ‚Äî the kind of arc a real app goes through over its first few months. See `JOURNEY.md` for the unfiltered account of what happened along the way.

## Run it

```bash
# 1. Install pg-flux (no Go required)
curl -sSfL https://raw.githubusercontent.com/nex-gen-tech/pg-flux/main/install.sh | sh

# 2. Set up the database
cp .env.example .env       # edit if your DSN differs
PGPASSWORD=pgflux createdb -h localhost -p 5440 -U pgflux pgflux_express_bookmarks

# 3. Apply the schema
set -a && source .env && set +a
pg-flux migrate apply

# 4. Install dependencies + run the API
npm install
npm run dev
```

The server starts on port `8012`. Then in another terminal:

```bash
# Create a user
curl -sX POST http://localhost:8012/users \
  -H 'Content-Type: application/json' \
  -d '{"email":"alice@example.com","handle":"alice"}' | jq

# Create a collection
curl -sX POST http://localhost:8012/users/<user-id>/collections \
  -H 'Content-Type: application/json' \
  -d '{"name":"Tech reads","color":"blue"}' | jq

# Save a bookmark
curl -sX POST http://localhost:8012/users/<user-id>/bookmarks \
  -H 'Content-Type: application/json' \
  -d '{
    "url": "https://example.com/article",
    "title": "An interesting article about PostgreSQL",
    "notes": "Great overview of GIN indexes"
  }' | jq

# List unread bookmarks
curl -s "http://localhost:8012/users/<user-id>/bookmarks?status=unread" | jq

# Full-text search
curl -s "http://localhost:8012/users/<user-id>/bookmarks/search?q=postgresql" | jq

# Mark as read
curl -sX PATCH http://localhost:8012/bookmarks/<bookmark-id> \
  -H 'Content-Type: application/json' \
  -d '{"status":"read"}' | jq

# Soft-delete
curl -sX DELETE http://localhost:8012/bookmarks/<bookmark-id>

# Check total bookmark count
curl -s http://localhost:8012/users/<user-id>/stats | jq
```

## What the schema demonstrates

The schema was built up across ~18 iterations against pg-flux ‚Äî intentionally different from the fastapi-todo example to exercise different pg-flux capabilities.

| Schema feature | File | Why it's here |
|---|---|---|
| UUID primary keys | `users.sql`, `bookmarks.sql` | `uuid DEFAULT gen_random_uuid()` ‚Äî different from bigserial |
| FK with `ON DELETE CASCADE` and `ON DELETE SET NULL` | `bookmarks.sql`, `collections.sql` | Both variants in the same table |
| `@renamed from=` hint | `users.sql` (username‚Üíhandle), `collections.sql` (title‚Üíname) | Two renames in the same build |
| `CHECK` constraint | `bookmarks.sql` (url format, notes length) | Auto-rewritten to `NOT VALID + VALIDATE` |
| `CREATE TYPE ... AS ENUM` | `types.sql` | `bookmark_status` with 4 values |
| `GENERATED ALWAYS AS ... STORED` | `bookmarks.sql` (`title_lower`, `search_vector`) | Two generated columns; tsvector for FTS |
| GIN index on tsvector | `bookmarks.sql` | `USING GIN (search_vector)` for `@@` queries |
| `jsonb NOT NULL DEFAULT '{}'` | `bookmarks.sql` (`metadata`) | Schemaless metadata bag |
| GIN index on jsonb | `bookmarks.sql` | `USING GIN (metadata)` for `@>` queries |
| Partial index with `ANY(ARRAY[...])` | `bookmarks.sql` (`idx_bookmarks_status`) | Was the B2 false-positive in fastapi-todo; now clean |
| Soft-delete pattern | `bookmarks.deleted_at` | Partial index keeps active queries fast |
| Views | `views.sql` | `unread_bookmarks` ‚Äî simple status filter |
| SQL functions | `functions.sql` | `count_user_bookmarks(uuid)` ‚Äî called from the API |
| PL/pgSQL trigger function + trigger | `triggers.sql` | DB-side `updated_at` ‚Äî different from app-level SET |
| Row-level security | `rls.sql` | Tenant isolation via `current_setting('app.user_id')` |

## The pg-flux workflow

```bash
# 1. Edit schema/*.sql
$EDITOR schema/bookmarks.sql

# 2. Generate the migration
pg-flux migrate generate --label my_change

# 3. Review the file in migrations/ ‚Äî it's plain SQL
$EDITOR migrations/20260521_*.sql

# 4. Apply
pg-flux migrate apply
```

For CI (note current caveats ‚Äî see JOURNEY.md):

```bash
pg-flux drift  --strict   # exit 2 if live DB != source (views always drift today)
pg-flux verify --strict   # exit 4 if live has undeclared objects (misses enums today)
```

## Notes

- The TypeScript interfaces in `src/types.ts` are hand-written to mirror `schema/*.sql`. pg-flux has first-party TypeScript codegen on the roadmap; until it lands, keep them in sync manually.
- The `search_vector` tsvector column and `metadata` jsonb column are excluded from API responses by default; add them to SELECT lists as needed.
- The `updated_at` column is maintained by the DB-side trigger `bookmarks_set_updated_at`, not by the app. No app code touches it.
- `DATABASE_URL` is used by both pg-flux and the app. Keep them pointing at the same DB.

## Reset to a clean DB

```bash
PGPASSWORD=pgflux dropdb   -h localhost -p 5440 -U pgflux pgflux_express_bookmarks
PGPASSWORD=pgflux createdb -h localhost -p 5440 -U pgflux pgflux_express_bookmarks
pg-flux migrate apply
```
