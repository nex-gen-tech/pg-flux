# fastapi-todo

A small FastAPI service backed by a **pg-flux-managed PostgreSQL schema**.

This example exists to show what an end-to-end stack looks like when pg-flux owns the schema and a Python app reads from it. It's the same pattern you'd use for any real service — the schema lives in `schema/*.sql`, migrations are generated and applied with pg-flux, and the app code talks to the resulting tables.

## Layout

```
fastapi-todo/
├── app/                  # FastAPI app
│   ├── main.py           # routes
│   ├── models.py         # pydantic models that mirror the SQL schema
│   └── db.py             # psycopg connection pool
├── schema/               # pg-flux source of truth — edit these
│   ├── users.sql
│   ├── todos.sql
│   ├── categories.sql
│   ├── tags.sql
│   ├── todo_tags.sql
│   ├── types.sql         # enums
│   ├── views.sql         # views
│   ├── functions.sql     # SQL functions
│   └── rls.sql           # row-level security policies
├── migrations/           # generated by `pg-flux migrate generate` — review, don't edit by hand
├── requirements.txt
├── .env.example
└── .pg-flux.yml
```

## Run it

```bash
# 1. Install pg-flux (no Go required)
curl -sSfL https://raw.githubusercontent.com/nex-gen-tech/pg-flux/main/install.sh | sh

# 2. Set up the database
cp .env.example .env       # edit if your DSN differs
createdb -h localhost -p 5440 -U pgflux pgflux_fastapi_example

# 3. Apply the schema
set -a && source .env && set +a
pg-flux migrate apply

# 4. Install Python deps + run the API
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload --port 8011
```

Then in another terminal:

```bash
# create a user
curl -sX POST http://localhost:8011/users \
  -H 'Content-Type: application/json' \
  -d '{"email":"alice@example.com","handle":"alice"}'

# create a todo
curl -sX POST http://localhost:8011/users/1/todos \
  -H 'Content-Type: application/json' \
  -d '{"title":"Buy milk","priority":"high"}'

# list active todos
curl -s http://localhost:8011/users/1/todos
```

## What the schema demonstrates

The schema in this example was built up across ~20 iterations against pg-flux — the kind of arc a real app goes through over its first few months. Every feature here exists because pg-flux handles it cleanly:

| Schema feature | File | Why it's here |
|---|---|---|
| FK with `ON DELETE CASCADE` | `todos.sql` | The everyday case |
| Partial indexes | `todos.sql` | `WHERE done = false`, `WHERE deleted_at IS NULL` |
| `@renamed from=` hint | `users.sql` | Renamed `username` → `handle` without data loss |
| Backfilled `NOT NULL` | `todos.sql` (`body`) | Added a default and the constraint together |
| `CHECK` constraint | `todos.sql` | Auto-rewritten to `NOT VALID + VALIDATE` by pg-flux |
| `CREATE TYPE ... AS ENUM` | `types.sql` | `todo_priority` used by `todos.priority` |
| `GENERATED ALWAYS AS ... STORED` | `todos.sql` (`title_lower`) | Computed column for case-insensitive search |
| Views | `views.sql` | `active_todos` filters out done + deleted |
| SQL functions | `functions.sql` | `count_user_todos(user_id)` — called from the API |
| Row-level security | `rls.sql` | Tenant isolation via `current_setting('app.user_id')` |
| Soft-delete pattern | `todos.deleted_at` | Partial index keeps active queries fast |

## The pg-flux workflow

You'll repeat these four commands forever:

```bash
# 1. Edit schema/*.sql
$EDITOR schema/todos.sql

# 2. Generate the migration
pg-flux migrate generate --label add_some_column

# 3. Review the file in migrations/ — it's just SQL
$EDITOR migrations/20260521_*.sql

# 4. Apply
pg-flux migrate apply
```

For CI:

```bash
pg-flux drift  --strict   # exit 2 if live DB ≠ source
pg-flux verify --strict   # exit 4 if live has undeclared objects
```

## Notes

- The Pydantic models in `app/models.py` are hand-written, not auto-generated. pg-flux's first-party codegen targets Go + TypeScript today; Python emitters are on the roadmap. Until they land, treat `models.py` as the contract you keep in sync with `schema/*.sql` by hand.
- The connection string used by both pg-flux and the app is `DATABASE_URL` in `.env`.
- The migrations in this example are real artifacts — generated against PostgreSQL 17 during the build of this app. You can `pg-flux migrate apply` them onto a fresh database to recreate the exact schema.

## Reset to a clean DB

```bash
dropdb   -h localhost -p 5440 -U pgflux pgflux_fastapi_example
createdb -h localhost -p 5440 -U pgflux pgflux_fastapi_example
pg-flux migrate apply
```
