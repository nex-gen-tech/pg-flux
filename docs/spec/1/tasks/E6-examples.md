## E6-T1 — Third comprehensive example app

**Summary:** Add a third first-party example that exercises a different domain, stack, and pg-flux features not covered by fastapi-todo (Python) or express-bookmarks (TypeScript). Target: a Go + chi (or gin) service — closes the loop on pg-flux being a Go-native tool and exercises `pg-flux gen --lang go` in a real app.

**Domain suggestion:** Multi-tenant event management (workspaces, events, attendees, tickets). This adds:
- Multi-schema tenancy (or `app.tenant_id` RLS pattern at scale)
- `FOREIGN KEY ... DEFERRABLE INITIALLY DEFERRED` (deferred FK for circular refs)
- `UNIQUE NULLS NOT DISTINCT` (PG15+)
- `GENERATED ALWAYS AS IDENTITY` (alternative to bigserial)
- A materialized view + `REFRESH MATERIALIZED VIEW CONCURRENTLY`
- `pg-flux gen --lang go` output wired into the app (not hand-written models)

**Acceptance criteria:**
- `examples/go-events/` directory with: schema, migrations, Go app, JOURNEY.md, README.md.
- All migrations generated by pg-flux and applied against a real DB.
- `pg-flux gen --lang go` output used directly in the Go app (not hand-written structs).
- JOURNEY.md follows the same format: what worked, real bugs hit, bottom line.
- Added to the `examples` CI job in `.github/workflows/test.yml`.
- `drift` and `verify` both exit 0 in CI.

**Dependencies:** E1-T1 (so no WORKAROUND comments are needed), E2-T1 (so verify covers triggers if any are used).
